# Part 10: Compose Features Support

## Goal

Bridge the gap between Part 8 (Docker Compose integration) and Part 9 (native features support) so that `dcx up` works for compose projects that declare `features` in `devcontainer.json`. Today such projects are blocked with `compose projects with features are not yet supported natively`. After this part, the full `test/` project (or any compose + features project) works end-to-end without the external `devcontainer` CLI.

## Architectural Approach

- Remove the hard guard in `internal/cli/up.go` that rejects compose projects when `cfg.HasFeatures()` is true.
- Teach `internal/devcontainer/compose.go` / `UpCompose` to build feature-augmented images before invoking `docker compose up`:
  1. Determine the base image for the target compose service. The base image can be specified directly in the compose file (`image:`), built from a Dockerfile within the compose file (`build:`), or inherited from another service (`extends:`). Start by supporting `image:` and `build:`; emit a clear error for `extends:` and other indirections.
  2. If `cfg.HasFeatures()`, invoke `features.BuildFeatureImage` with that base image reference (same API as the native image/Dockerfile path). This produces a stable deterministic tag (e.g. `dcx-<project>-feat:<hash>`).
  3. Inject the feature-augmented image into the compose override file (`dcx.compose.override.yml`) so that the target service uses the augmented image instead of the original base image. The override is already generated by `writeComposeOverride`; add an `image:` field when a feature image is present.
  4. Pass the `--upgrade-lockfile` flag through to `BuildFeatureImage` so that compose projects can also re-resolve feature digests.
- Handle service discovery edge cases:
  - If the compose file declares `build:` (Dockerfile-based service), the feature layer is applied *after* that Dockerfile build. Reuse `BuildImage` (Part 4) to build the service image if needed, then pass the resulting tag as the base image to `features.BuildFeatureImage`.
  - If the compose file declares `image:` directly, use that reference as the base image for feature layering.
  - Cache the final feature-augmented image using the same stable-tag logic as the native path so that repeated `dcx up` calls are fast.
- Keep changes minimal in the compose path:
  - The existing override file generation (labels, env, mounts, keep-alive entrypoint) stays unchanged.
  - The existing `docker compose up` invocation stays unchanged; only the temporary override file gains an optional `image:` override.
- Add a compose-with-features test project under `test/` or a temporary fixture so CI can verify the path.

## Acceptance Criteria

- [ ] `dcx up` on a compose project that specifies `features` in `devcontainer.json` succeeds instead of returning the hard error.
- [ ] The target compose service container is created from a feature-augmented image that includes all resolved features.
- [ ] Re-running `dcx up` without changes reuses the cached feature-augmented image (fast path).
- [ ] `dcx up --upgrade-lockfile` on a compose project re-resolves feature digests and writes a new `devcontainer-lock.json`.
- [ ] `dcx down` and `dcx stop` on the resulting compose project work correctly.
- [ ] `dcx exec` on the resulting compose project opens a shell in the feature-augmented container.
- [ ] Unit tests verify that the compose override file contains the expected `image:` override when features are present.
- [ ] `go test ./... -race`, `go vet ./...`, and `gofmt -l .` pass.
